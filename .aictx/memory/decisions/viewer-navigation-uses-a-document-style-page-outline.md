The local memory viewer sidebar and memory page should feel like a quiet document workspace rather than a maintenance cockpit. Use a document-style page outline with workspace pages first, saved memory views second, clear active states, and secondary write/export controls tucked behind quieter disclosure UI. Keep the Project Memory page calm: a slim explanatory summary, compact toolbar, and memory library should be the default view. Selecting a memory should expand that memory inline under its row as a document toggle, not open a side inspector or a separate page. The row itself is the open/close control; avoid duplicate Open/Close CTAs in the expanded content. Rows should look like page/database rows rather than heavy cards: minimal borders, subtle hover fill, muted icons, no blue browser focus outline, and quiet section headings. Inline expanded content should include Memory content open by default, provenance and direct relation rows as toggles, and technical JSON details behind a toggle. Do not show a per-memory relation graph by default; for individual memories, relation rows are clearer than tiny node diagrams. A future graph should be a separate project/section-level Memory Map only if relation density makes it useful. Following a related memory should reveal the target row in the library before expanding it. Keep the viewer focused on human inspection of project memory; do not make navigation imply editing, repair, or automated maintenance as the default path.
